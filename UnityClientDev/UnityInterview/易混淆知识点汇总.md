## 1) 值类型 vs 引用类型（C# 与 Unity 场景中的常见混淆）

- 核心差异：值类型在栈或内存上分配后按值复制；引用类型复制的是对同一对象的引用。Unity 中的对象还涉及到托管内存与原生内存的混用场景。
- 常见误区：把引用类型变量简单当成值类型复制导致多处修改同一对象的副作用；或者错误地把结构体当作引用类型处理，产生复杂的状态变更。
- 区别要点：
  - int、float、struct 等为值类型；class、string、array 等为引用类型（注意 string 也是引用类型但表现特殊）
  - 赋值时，值类型复制值；引用类型复制引用，修改其中一个会影响其他引用。
- 练习：
  1) 定义一个简单的结构体 Point { public int x,y; }，演示赋值后修改副本不影响原始值。 
  2) 定义一个简单的类 Player { public int health; }，创建两个实例并让一个引用指向另一个，修改其中一个的 health，观察另一端的变化。
  3) 在 Unity 场景中创建一个 ScriptableObject 与一个 MonoBehaviour 作为数据载体，比较引用传递与值传递在剪切/拷贝时的差异。

---------------------------------------------------------------------

## 2) 单例模式与静态/全局状态

- 阶段0中的单例常用于数据管理、全局状态；阶段4进一步区分 MonoBehaviour 单例与纯静态单例的使用场景及风险。
- 区别要点：
  - 单例是一种设计模式，旨在保证一个类只有一个实例，并提供全局访问点；静态类则是没有实例、全局唯一的类，直接通过类名访问成员。
  - MonoBehaviour 单例受 Unity 生命周期影响，需要在场景中创建对象或通过脚本实例化；纯静态单例没有生命周期，适合纯数据/工具类。
- 练习：
  1) 实现一个简单的静态工具类 MathUtil，提供常用数学方法；再实现一个 MonoBehaviour 单例的 AudioManager，演示在场景切换时的稳定性与清理策略。
  2) 设计一个基于 MonoBehaviour 的通用 Singleton<T> 基类，确保线程安全（简化版）并适配 Unity 场景对象。

## 3) 委托、事件与 UnityEvent

- 区别要点：
  - 委托是类型安全的函数指针；事件是在委托基础上的发布-订阅机制，提供更好的封装和解耦。
  - UnityEvent 是 Unity 专有的可序列化事件，便于在 Inspector 配置订阅，适合 UI 与编辑器友好场景。
- 练习：
  1) 写一个简单的委托和一个事件示例，模拟玩家血量变化时通知 UI 与声音系统。
  2) 用 UnityEvent 重构上面的事件，使其在 Inspector 中可绑定回调。

## 4) 资源加载模型：Resources、AssetBundle、Addressables

- 区别要点：
  - Resources：简单但不推荐，易造成体积膨胀和内存管理困难；适合小资源、快速原型。
  - AssetBundle：打包资源、按需加载，适合热更新与分包加载。
  - Addressables：推荐方案，自动引用计数、依赖管理、异步加载，适合大规模资源。 
- 练习：
  1) 设计一个资源加载管理器，支持 Resources.Load、AssetBundle 读取与 Addressables 的基本加载，比较三个 API 的使用场景。
  2) 实现一个简单的资源缓存策略，避免重复加载。

## 5) 协程 vs async/await

- 区别要点：
  - 协程是 Unity 特有的在主线程上的分时执行，不是真正的并发；async/await 是 C# 的异步编程，底层基于 Task，能跨线程执行（受限于 Unity API 的线程安全性）。
  - 协程更适合等待场景、动画、逐步流水等状态机驱动的流程；async/await 适合 I/O、网络请求等（可用 Task.Run 包裹 CPU 密集型工作）。
- 练习：
  1) 用协程实现一个加载进度条；2) 用 async/await 模拟从网络获取数据并在主线程应用结果。

## 8) 进程 vs 线程（操作系统基础）

- 关联点：Unity 多线程（线程池、Job System）与操作系统进程/线程模型的关系
- 核心区别：

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源拥有 | 独立地址空间、文件句柄等 | 共享所属进程的资源 |
| 通信方式 | IPC（管道、消息队列、共享内存） | 直接读写共享内存（需同步） |
| 切换开销 | 大（需切换页表、刷新TLB） | 小（仅保存/恢复寄存器） |
| Unity 中的体现 | 每个 Unity 实例是一个进程 | Job System、Task 运行在线程上 |
| 应用场景 | 游戏多开、编辑器模式下运行游戏预览 | 多线程渲染、异步资源加载 |

- C++ 对比：`CreateProcess()` vs `std::thread`，C++ 无内置进程管理（需 Windows API）
- 面试提示："进程隔离，线程共享；进程间通过 IPC 通信，线程间通过共享内存加锁"

---------------------------------------------------------------------
## 9) TCP vs UDP（计算机网络基础）

- 关联点：Unity 多人游戏（Mirror/Photon）、网络同步方案的选择
- 核心区别：

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（重传、确认） | 不可靠（可能丢包） |
| 顺序 | 保序 | 不保序 |
| 头部开销 | 20-60 字节 | 8 字节 |
| Unity 中的应用 | 角色登录、聊天系统、匹配服务 | 实时同步（位置、输入）、帧同步 |
| C# API | TcpClient/TcpListener | UdpClient |
| C++ API | socket(SOCK_STREAM) | socket(SOCK_DGRAM) |

- 面试提示："TCP 用时间换可靠性，适合状态同步；UDP 速度快但有损耗，适合帧同步"

---------------------------------------------------------------------
## 10) 栈 vs 堆（内存管理基础）

- 关联点：值类型（栈）vs 引用类型（堆）、GC 负担、性能优化
- 核心区别：

| 维度 | 栈（Stack） | 堆（Heap） |
|------|-------------|------------|
| 分配速度 | 极快（仅移动SP指针） | 慢（需查找合适内存块） |
| 管理方式 | 自动（函数调用结束弹出） | 手动/C# GC 自动回收 |
| 大小限制 | 很小（默认 ~1MB/线程） | 很大（可达数GB） |
| 存储内容 | 局部变量、参数、返回地址 | 动态分配的对象、全局数据 |
| 数据特点 | LIFO 顺序，连续内存 | 随机分配，可能碎片化 |
| C# 中的使用 | struct、局部值类型变量 | class、string、数组 |
| C++ 对比 | 自动管理，无需操心 | new/delete、智能指针管理 |
| Unity 优化 | 避免 struct 频繁装箱 | 对象池减少 GC 分配 |

- 面试提示："栈是自动管理的高效临时存储，堆是灵活但需管理的长期存储——理解这个关系是做 Unity 性能优化的基础"
