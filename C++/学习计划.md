## 第一阶段：基础入门（3 天）

### 目标
理解 Unity 的**帧循环**本质上是什么，输入系统底层怎么工作

### 第 1 天：Hello Window — Unity 的生命周期

**Unity 对照：**

```
Unity:                Raylib:
Awake()  / Start()  → InitWindow() + 初始化代码
Update()             → while (!WindowShouldClose())
  Time.deltaTime     →   GetFrameTime()
  Debug.Break()      →   WindowShouldClose()
OnDestroy()          → CloseWindow()
```

Unity 的 `Update()` 不是魔法——它就是一个 `while` 循环每帧调用你的代码。每个 `MonoBehaviour` 的 `Update()` 背后，引擎都在做你 while 循环里的事。

**C++ 深度：**
- `int main()` 是程序的入口——所有 C++ 程序都从这里开始
- `#include "raylib.h"` 做了什么？它把 raylib.h 的全文**复制粘贴**进来（预处理阶段）
- 函数定义 vs 声明：`#include` 拿到的是声明（告诉编译器函数存在），定义在 `.lib` 中（链接阶段找到）

**检查点：** `BeginDrawing()/EndDrawing()` 就是 Unity 的渲染管线开关——期间所有绘制指令被统一提交到 GPU。

### 第 2 天：动画与物理 — Unity 的 Time.deltaTime
*（今晚目标：理解 dt + 修改物理参数，45 分钟包过）*

**Unity 对照：**

| Unity | Raylib | 说明 |
|-------|--------|------|
| `Time.deltaTime` | `GetFrameTime()` | 上一帧耗时（秒） |
| `Time.time` | `GetTime()` | 总运行时间 |
| `Time.fixedDeltaTime` | 你自己写 `dt` 累加 | Unity 物理步进 |
| `FixedUpdate()` | 固定时间步循环 | 物理稳定性保证 |

为什么 Unity 有 `Update()` 和 `FixedUpdate()` 两个？因为物理需要**固定时间步**才能稳定——你在 02 中可以亲手体验不同帧率下的物理差异。

**C++ 深度：**
- `float` 为什么加 `f` 后缀？`float x = 1.0;` 会引发 narrowing conversion 警告——C++ 是强类型语言
- `int` vs `float`: C++ 不会自动帮你做"合理"的类型转换，这是它和 C# 的重要区别
- `GetFrameTime()` 返回的是 `float`——Unity 中 `Time.deltaTime` 也是 32 位浮点

- [ ] 理解 `速度 × dt = 位移` —— 这就是 Unity 中所有 `transform.position += velocity * Time.deltaTime` 的底层
- [ ] 练习题：去掉 `dt` 看看物理行为怎么变化（和低帧率下的 Unity 行为一致）

### 第 3 天：输入处理 — Unity 的 Input 系统

**Unity 对照：**

| Unity | Raylib | 说明 |
|-------|--------|------|
| `Input.GetKey(KeyCode.D)` | `IsKeyDown(KEY_D)` | 按住时每帧 true |
| `Input.GetKeyDown(KeyCode.Space)` | `IsKeyPressed(KEY_SPACE)` | 按下瞬间 true，下次自动 false |
| `Input.mousePosition` | `GetMousePosition()` | 鼠标位置 |
| `Input.GetAxis("Horizontal")` | 手动检测方向键 | Unity 帮你做了死区和平滑 |

Unity 的 `Input.GetKeyDown()` 为什么只触发一次？因为引擎内部维护了一个"上一帧按键状态"的 bool 数组，`GetKeyDown = current && !previous`。你自己可以用同样的模式实现。

- `enum` vs `enum class`：`enum class Direction { LEFT, RIGHT }` 是 C++11 的类型安全枚举——不能隐式转 int，不会污染命名空间
- `Clamp()` 函数：Unity 的 `Mathf.Clamp` 也是同样的实现，只是藏在引擎 DLL 里你看不到

## 第二阶段：C++ 进阶（3 天）

### 第 4 天：类与对象 — Unity 的 GameObject 底层

**Unity 对照：**

Unity 的 `GameObject` 是一个容器，挂载各种 `Component`。Raylib 没有 GameObject——你需要手动创建 struct/class 来模拟它。

这个项目让你亲手写下 Unity 帮你做的事情：
- `Player p;` 在栈上分配——Unity 的组件在堆上由引擎管理
- `std::vector<Player>` 动态数组——Unity 的 `FindObjectsOfType` 背后也是一个类似的列表

**C++ 深度：**
- `class` 默认 `private`，`struct` 默认 `public`——这是 C++ 中两者唯一的区别
- 构造函数：C++ 保证对象创建时构造函数被调用——Unity 的 `Awake()` 也是类似的生命周期保证
- `std::vector` 扩容机制：当容量不够时，分配新内存 → 拷贝/移动旧元素 → 释放旧内存。这是理解 `List<T>` 底层的关键
- `this` 指针：每个成员函数都隐式带一个 `this` 参数——C++ 对象模型的核心

```cpp
// 编译器把 p.GetX() 变成：
// getX(&p)  —— 第一个参数是对象地址

// 所以成员函数本质上就是：
float getX(Player* this) { return this->x; }
```

### 第 5 天：精灵与动画 — Unity 的 SpriteRenderer + 帧动画

**Unity 对照：**

Unity 的 SpriteRenderer 内部做的是：
1. 获取 Sprite 的 Texture2D
2. 计算 UV 坐标（你写的 `source` Rectangle）
3. 发送到 GPU 渲染

`DrawTexturePro(texture, source, dest, origin, rotation, tint)` 就是 Unity SpriteRenderer 的底层实现。在 Unity 中你调整 Sprite 的 pivot、rotation、scale——在 Raylib 中这些就是 `origin`、`rotation`、`dest.width/height` 参数。

**C++ 深度：**
- `struct Rectangle { float x, y, width, height; };` —— 这是 C 风格 POD（Plain Old Data），内存布局就是 4 个 float 连续排列
- 传值 vs 传引用 vs 指针：`DrawTexturePro(..., rotation, color)` 传 float 值（4 字节拷贝），`source` 传 Rectangle 值（16 字节拷贝）
- C++ 结构体按成员顺序连续存放在内存中——这在 Unity 的 `StructLayoutAttribute` 中可控制，在 C++ 中是默认行为

### 第 7 天：音频 — Unity 的 AudioSource 底层

**Unity 对照：**

| Unity | Raylib | 说明 |
|-------|--------|------|
| `AudioSource.Play()` | `PlaySound()` | 播放一次性音效 |
| `AudioSource.clip` | `LoadSound()` | 加载音频到内存 |
| `AudioSource.volume` | `SetSoundVolume()` | 0.0-1.0 |
| `AudioSource.pitch` | `SetSoundPitch()` | 改变播放速度 |
| `AudioSource.loop = true` | `PlayMusicStream()` + 循环 | 流式长音频 |

在 Unity 中你拖一个音频文件到 AudioSource 上——在 Raylib 中你要手动调用 `LoadSound`、`PlaySound`、`UnloadSound`。Unity 帮你管理了资源的生命周期，Raylib 让你自己掌控。

**C++ 深度：**
- 手动生成的波形 PCM 数据是**未压缩的音频**——Unity 中的 `.wav` 也是同样的 PCM 格式
- `InitAudioDevice` / `CloseAudioDevice`：对应 Unity 中 AudioSettings 初始化和销毁
- 资源加载和卸载必须成对出现——C++ 没有垃圾回收，忘记 `UnloadSound` 就是**内存泄漏**
- 这也解释了为什么 Unity 的 `Resources.UnloadUnusedAssets` 在某些时候不生效——你需要在合适的时机调用

### 第 12 天：摄像机 — Unity Camera + Cinemachine 的底层
*（今晚目标：跑 08，理解四个参数，45 分钟足够）*

**08_2DCamera → Unity 的 Camera 和 Cinemachine：**

```cpp
Camera2D camera = { 0 };
camera.target = playerPos;      // Cinemachine 的 Follow Target
camera.offset = screenCenter;   // Cinemachine 的 Screen X/Y
camera.zoom   = 1.0f;           // Camera.orthographicSize
camera.rotation = 0.0f;         // Camera.transform.rotation
```

Cinemachine 的"平滑跟随"本质：
```cpp
// 每帧让 camera.target 向 player 靠近一定百分比：
camera.target.x += (player.x - camera.target.x) * 5.0f * dt;
// 这就是 Lerp 的原理——Unity 中 Vector3.Lerp 的内部实现
```

**C++ 深度：**
- `Camera2D camera = { 0 };` 是 **zero-initialization**——所有字段初始化为 0，Unity 中的 `new Vector3()` 同理
- `expf(logf(zoom) + wheel * 0.1f)` 使用对数缩放——因为**缩放是乘法关系，线性加减不自然**。Unity 的 `Camera.orthographicSize` 也是对数感知
- `std::clamp()` vs 手写 if：C++17 标准库函数，Unity 的 `Mathf.Clamp` 也一样

### 第 14 天：缓动动画 — Unity AnimationCurve 的数学公式

**10_EasingsAnimation → Unity 的 AnimationCurve：**

```cpp
ballPositionX = EaseElasticOut(time, start, change, duration);
```

在 Unity 中，你在 Animation 窗口拖出一条曲线，引擎内部就生成了一个类似的函数。DOTween 的 `.SetEase(Ease.OutBounce)` 调用的也是同样的数学公式。

| Unity | Raylib | 说明 |
|-------|--------|------|
| `AnimationCurve.Evaluate(t)` | `EaseCubicOut(t, b, c, d)` | 输入时间 → 输出插值 |
| `DOTween.SetEase(Ease.OutElastic)` | `EaseElasticOut()` | 弹簧效果 |
| DOTween 的 Duration 参数 | `d` (持续时间) | 动画时长 |

**C++ 深度：**
- 所有这些缓动函数都有相同的签名：`float func(float t, float b, float c, float d)`——这就是**函数指针**的基础
- `#include "reasings.h"` 中所有函数都是 `static inline`——这意味着函数的代码被直接嵌入到调用处，没有函数调用开销（Unity 的 Burst Compiler 也做同样的事）
- `rand()` 在 10 中控制循环计数，在 11 中控制粒子方向——**伪随机**在游戏中无处不在

### 第 15 天：粒子系统 — Unity Particle System 的核心里面

**11_ParticleSystem → Unity 的 Particle System 组件：**

Unity 的 Particle System 有几千个参数——但它内部核心逻辑和你写的 11 一样：
1. 每帧发射新粒子（Emit）
2. 更新每个粒子的位置/速度/颜色/大小（Update）
3. 移除死亡的粒子（Clean）
4. 绘制所有存活粒子（Draw）

```cpp
struct Particle {
    Vector2 position;   // Unity: particle.position
    Vector2 velocity;   // Unity: particle.velocity
    float radius;       // Unity: particle.startSize
    Color color;        // Unity: particle.startColor
    float lifeTime;     // Unity: particle.remainingLifetime
    bool alive;         // Unity: particle.lifetime <= 0
};
```

**C++ 深度：**
- **循环缓冲区**：`head` 和 `tail` 指针在环形数组中追逐——避免频繁 `new`/`delete`。Unity 的 ParticleSystem 内部也用了对象池模式
- `rand() % 360` 生成 0-359 的随机角度——但 `rand()` 的分布不均匀。Unity 的 `Random.Range` 用了更好的算法
- 注意 `Particle` 是**栈上数组** `buffer[MAX_PARTICLES]`——这是编译时分配，没有运行时开销。Unity 的粒子系统也在内部维护一个预分配数组

### 第 16 天：渲染纹理 — Unity RenderTexture 亲手实现
*（今晚目标：跑 12 理解离屏渲染流程，30 分钟搞定）*

**12_RenderTexture → Unity 的 RenderTexture：**

Unity 中你创建一个 `RenderTexture` 赋值给 Camera.targetTexture，摄像机的画面就会被渲染到纹理上而不是屏幕。Raylib 中：

```cpp
// 创建渲染目标
RenderTexture2D target = LoadRenderTexture(300, 300);

// 渲染到纹理（不是屏幕）
BeginTextureMode(target);
    ClearBackground(SKYBLUE);
    DrawCircleV(ballPos, ballRadius, MAROON);
EndTextureMode();

// 把纹理当普通贴图用
DrawTexturePro(target.texture, ...);
```

这个流程和 Unity 完全一致——当你在 Unity 中做**手机屏幕里的监控画面、镜子效果、小地图**时，背后都是同一个原理。

**C++ 深度：**
- `DrawTexturePro` 的 `source.height` 传负值翻转 Y 轴——因为 OpenGL 纹理原点在左下角，而渲染纹理原点在左上角。Unity 帮你自动处理了这个差异
- `RenderTexture2D` 包含 `texture`（GPU 纹理）和 `depth`（深度缓冲）——Unity 的 RenderTexture 也是同样的结构

### 第 17 天：程序化纹理 — Unity Texture2D 的 CPU 生成

**13_ImageGeneration → Unity 的 Texture2D 程序化生成：**

```cpp
// 在 CPU 上生成图像
Image noise = GenImagePerlinNoise(width, height, 50, 50, 4.0f);
// 上传到 GPU
Texture2D tex = LoadTextureFromImage(noise);
// 释放 CPU 内存（GPU 已有一份）
UnloadImage(noise);
```

Unity 中等效操作：
```csharp
Texture2D tex = new Texture2D(width, height);
tex.SetPixels(colors);    // CPU 侧设置像素
tex.Apply();              // 上传到 GPU（= LoadTextureFromImage）
```

**C++ 深度：**
- 纹理在**GPU 显存**中，Image 在 **RAM** 中。`UnloadImage` 释放 CPU 内存，`UnloadTexture` 释放 GPU 内存——这和 Unity 完全一致
- 9 种纹理全部用 `GenImage*` 生成，不需要任何外部图片文件——这就是"程序化内容"的含义
- 注意代码中 `UnloadImage` 在 `while` 循环之前就调用了——Image 数据上传到 GPU 后，CPU 侧数据即可释放

## 学习方法建议

### 核心原则

> **每个 Raylib 项目都是一个"拆开的 Unity"——你在 Raylib 中亲手写的每一行代码，Unity 内部都有一套对应的实现。**

### 晚间学习节奏（22:00 后）

```
22:00 ─ 22:05  回顾昨天的笔记（5分钟）
22:05 ─ 22:25  运行 + 读代码（20分钟核心）
22:25 ─ 22:35  动手小改 + 观察效果（10分钟）
22:35 ─ 22:45  思考：这和 Unity 的什么对应？（10分钟对标）
22:45         停！关电脑，让大脑离线
```

**累了保底模式：** 改一个数字（颜色/速度/大小），写一行笔记，10 分钟收工。连续打卡比强度重要 10 倍。

### 遇到问题怎么办

1. 先看编译器错误信息 — C++ 的错误信息比 C# 精确得多，习惯它
2. 注释掉问题代码，逐步缩小排查范围
3. 对比示例代码
4. 搜索 `raylib [你的问题]`
5. 问人 — Discord (discord.gg/raylib) 或问 AI
